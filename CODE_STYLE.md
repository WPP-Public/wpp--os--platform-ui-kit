# Code style guide

## Generic
- Leverage the Shadow DOM to encapsulate styles
and structure, avoiding global style leakage.
```tsx
@Component({
  tag: 'wpp-component',
  shadow: true
})
```
- Use the <slot> element to allow content
projection within your shadowed components.
```tsx
render() {
    return (
        <div>
            <slot></slot>  // Placeholders for user-provided content
        </div>
    );
}
```
- Add description about all the slots and part attributes
```tsx
/**
 * @slot - Should contain a list of `wpp-option` elements that represents the current options list.
 *
 * @part input - input element
 */
@Component({
  tag: 'wpp-component',
  shadow: true
})
```
- Ensure that the component has minimal dependencies, making it
lightweight and efficient. External libraries should be carefully vetted.
- Avoid hardcoding strings or numbers. Use constants or properties instead,
making the code more maintainable and clear.

## Naming conventions
- **Components**: Use kebab-case for component tags and file names.
E.g., <wpp-accordion>, wpp-accordion.tsx.
- **Properties and Methods**: Use camelCase. E.g., withDivider, wppChange().

## Code organization in the component
The source file of the component should be organized in the same way
for all the components, in order for consistency and easy of work with
the components. Provided below the component example:
```tsx
export class WppComponent {
  /**
   * 1. Own properties of the component.
   * They are not exposed and used only for the internal purposes.
   */
  private detail = 'detail'

  /**
   * 2. Reference to host HTML element.
   */
  @Element() host: HTMLWppComponent

  /**
   * 3. State() variables
   */
  @State() maxHeight: number = 100
  @State() toggleOverflow: boolean = false

  /**
   * 4. Properties
   */
  @Prop({ reflect: true }) readonly withDivider: boolean = true
  @Prop({ reflect: true, mutable: true }) expanded: boolean = false

  /**
   * 5.Events section
   */
  @Event({ bubbles: false, composed: false }) wppChange: EventEmitter<ComponentChangeEventDetail>
  @Event({ bubbles: false, composed: false }) wppFocus: EventEmitter<FocusEvent>

  /**
   * 6. Watch section
   */
  @Watch('loading')
  onLoadingChange(loading: boolean) {}

  /**
   * 7. Listen secion
   */
  @Listen('wppItemChange', { capture: true })
  onItemChange() {}

  /**
   * 8. Component lifecycle events
   * Ordered by their call order.
   */
  connectedCallback() {}
  disconnectedCallback() {}

  /**
   * 9. Public methods API
   */
  @Method()
  async open(): Promise<boolean> {}

  /**
   * 10. Local methods
   */
  getHostClassNames() {}

  /**
   * 11. Render function
   */
  render() {}
}
```

## Properties
- Always provide sensible default values for properties.
- Avoid mutable properties unless there's a clear use case.
```tsx
@Prop({ mutable: true })
```
- Define the Property Type Explicitly.
- Always document props. Use JSDoc comments to describe the prop,
its purpose and default behavior if applicable.
```tsx
/**
 * If the modal can be closed by clicking outside of it.
 */
@Prop() readonly disableOutsideClick: boolean = false
```
- Prefix boolean properties with `is` or `with`.
```tsx
@Prop() isActive: boolean = false;
@Prop() withDivider: boolean = false;
```
- Reflect Props To Attributes only in tha case If a prop should be
observable from the outside (e.g., for styling or accessibility reasons).
```tsx
@Prop({ reflect: true }) checked: boolean = false;
```
- Handle Prop Changes. If you need to perform side effects when a prop changes, use the watch decorator.
```tsx
@Prop() value: string;

@Watch('value')
onValueChanged(newValue: string, oldValue: string) {
  console.log(`Value changed from ${oldValue} to ${newValue}`);
}
```

## Event handling
- Always name the event with the `wpp` prefix. For example `wppChange`, `wppFocus`
- Create Non-Bubbling and Non-Composing Custom Events by default
```tsx
@Event({ bubbles: false, composed: false }) wppFocus: EventEmitter<FocusEvent>
```

## Styles
- Always utilize the Shadow DOM for your components.
It ensures style encapsulation, which means your component's
styles won't bleed out, and external styles won't affect your component.

- Use the `:host` pseudo-class to target the main component itself.
It provides a way to style the root element of your component.
```css
:host {
  display: block;
}
```
You can also use conditional styles:
```css
:host(.hidden) {
  display: none;
}
```

- Expose Styles with `:part:` If you wish to allow external customization of
certain internal elements in your component, use the ::part pseudo-element.
```tsx
render() {
  return <div part="header">Header Content</div>;
}
```
```css
wpp-component::part(header) {
  font-weight: bold;
  color: blue;
}
```
This allows limited styling from outside the component without breaking encapsulation.

- Semantic Naming: Always use meaningful, semantic class and part names.
Avoid ambiguous names like btn. Instead, use names like submit-button to make it clear.

- Avoid Deep Nesting: Excessive specificity can make styles hard to override if needed.
Keep your selectors as flat as possible

- Use CSS Custom Properties (variables) to allow easy theming.
More details about theming can be checked in the Storybook.
<br/>All design tokens names in WPP Design System are created by this pretty simple rule

`--wpp-button-primary-bg-color-hover` --> `--[ds-prefix]-[component-name]-[component-variant](?)-[style-name]-[modifier](?)`

## Internationalization (i18n)

Localization is a crucial part.
We need to omit all the hardcoded text and it should be
replaced with properties to provide locale-specific content.

Example of defining localizations:
```tsx
@Prop() readonly locales: ComponentLocales = {
  nothingFound: 'Nothing found',
  more: 'more',
  selected: count => `${count} selected`,
  loading: 'Loading...',
  createNewElement: 'Create new element',
}
```

The locales property provides default translations for a component.
Users can override these values as needed. Here's how to
utilize the 'loading' translation in a component:

```tsx
<wpp-list-item class="loading">
  <wpp-spinner slot="left" />
  <wpp-typography type="s-body" slot="label">
    {this.locales.loading}
  </wpp-typography>
</wpp-list-item>
```

## Accessibility

- Always use appropriate semantic elements. These provide native
accessibility benefits and make it easier for assistive
technologies to understand the content.
- Pay attention to focus order and ensure it's logical and intuitive.
- Use ARIA (Accessible Rich Internet Applications) attributes
judiciously to enhance accessibility when semantic HTML doesn't suffice.
```tsx
<input aria-label="Search" type="text" />
```
